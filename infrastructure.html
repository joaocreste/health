Infrastructure & ETL workflow

End-to-end architecture of the JC Advisory health webapp — how raw exports from wearables, scales, blood-pressure cuffs, CGMs, DICOM workstations and lab PDFs become charts, AI insights and bilingual narrative pages. Static-build pipeline runs locally; live runtime is fully Cloudflare-native with a Neon Postgres database (Frankfurt) and the Anthropic Claude API for ingestion and chat.

03 · Data sources

Where the raw signal comes from

All inputs live under Patients/<Patient>/ at the repository root. The ETL script accepts a JC_DATA_DIR environment variable so a different patient folder can be swapped in without touching the code. Bytes that are too large to commit (DICOM exports, Apple Health full XML, audio transcripts) are gitignored locally and live only on the engineer's laptop until they have been processed into smaller artifacts.

04 · Build-time ETL

Python pipeline · bin/extract.py

A single Python script reads every CSV / JSON / PDF under the patient folder, aggregates per-day series, computes Tukey box-statistics + percentiles, and writes two artifacts back into the web bundle. The script is idempotent — running it twice with the same inputs produces byte-identical outputs (apart from the timestamp comment). It is invoked manually after each new export lands on the laptop.

ETL flow · raw exports → aggregated JS + JSON
SVG · 1080 × 480

Inputs (read paths)

  • Resolved path: Patients/Joao Victor Creste/ by default; override with JC_DATA_DIR
  • Subfolder convention: Oura/App Data/, Apple Health/electrocardiograms/, Withings/, plus root-level manual CSVs
  • Missing files: are tolerated — the script logs the gap and continues
  • Format quirks: Oura uses semicolon-separated CSVs with JSON-encoded sub-objects in single cells; the script parses these inline

Outputs (written into web/)

  • web/assets/data.js — single-file JS module exporting top-level constants (WEIGHT, HRV_RHR, STEPS, etc.) consumed directly by the static pages
  • web/assets/metrics.json — every computed summary metric, keyed by name, for one-shot reads
  • Stdout summary: a console table of the headline numbers so the engineer can verify before deploying
  • Manual sources panel: reports which small CSVs are present vs empty (templates)
When to re-run. After every new export from Oura, Apple Health, Withings or the CGM; after editing any manual CSV; and after any change to bin/extract.py itself. Then commit the regenerated data.js / metrics.json in the same commit as any narrative refresh — the two are bound together.
05 · Runtime path

What happens when a user clicks a link

Static pages are served straight from Cloudflare Pages — no Worker invocation, no database read. The Worker only wakes up for /api/* routes: chat, dashboards, ingestion, the admin console. This split keeps the static-data path cheap (CDN cache hits) and quarantines all data-sensitive logic to a single file (web/_worker.js, ~1,200 LOC).

Request lifecycle · static vs. /api/* split
SVG · 1080 × 500

Static path (95% of traffic)

  • What: HTML pages, data.js, metrics.json, images, fonts
  • Where: served from Cloudflare's global edge cache
  • Cost: effectively zero per request
  • Latency: typically < 50 ms anywhere in the world

Worker path (data + AI)

  • What: auth, dashboards, ingestion, AI chat, admin actions
  • Where: Cloudflare Workers runtime (V8 isolates)
  • Boot: ~5 ms cold start per region
  • Limits: 30-second CPU max per request; the chat stream stays inside that

Patient routing

  • Client side: web/assets/patient-context.js reads ?patient= or session, renders the right view per patient
  • Patient Zero (Joao): static hand-crafted HTML
  • Bespoke patients (Paulo, Silvana, Leo): per-patient renderers in the same file
  • Generic patients: data-driven from /api/patient-summary
06 · Database structure

Neon Postgres — schema, conventions, and the structured / unstructured split

31 tables in 10 domains, defined in a single Drizzle file (db/schema.ts, 591 LOC). The schema is deliberately small and patient-centric: every clinical row carries a patient_id foreign key with ON DELETE CASCADE, so a single patient deletion walks the entire graph cleanly — a hard requirement for GDPR / LGPD right-to-erasure. Bulky binaries (PDFs, DICOM, audio) live in R2; the database stores the R2 key, not the bytes.

Three storage shapes — sometimes combined in the same row

The platform mixes three patterns across its 31 tables: strict typed columns for facts that fit a schema, JSONB payloads for shape-flexible structured data, and R2 blob keys for content that should never live inside the database. Most tables combine two or three of these patterns in the same row.

Three storage patterns · how a row is shaped
SVG · 1080 × 540
source_blob_key on most clinical tables (provenance link to original PDF) Combined-pattern reality A typical row carries all three: typed columns for the facts you query (patient_id, name, dose), an optional JSONB payload for instrument-specific data (payload, metadata, extras), and an optional source_blob_key pointing back to the original PDF or CSV in R2 for audit / re-parsing. The schema chooses pattern by query shape: structured when you plot it, JSONB when the shape varies per row, R2 when the bytes shouldn't bloat the database.

Conventions enforced across the schema

Identity & access invariants

  • Patient ownership. Every clinical row has patient_id uuid not null references users.id on delete cascade — a single patient delete walks the graph.
  • Reference data exception. psych_dimensions is the only patient-independent table (13 AMPD axes, seeded by a migration).
  • Time always with zone. timestamptz everywhere; never naive timestamps. date used only when no clock-time information exists upstream (e.g. started_at on a medication).
  • UUID v4 primary keys for general rows; bigint identity for high-volume append-only tables (glucose_points, audit_log).
  • Provenance. Most rows carry source_blob_key text pointing back to the original document in R2 — so a row can always be re-parsed against the original source.

Indexing & constraints

  • Patient-time composite index on every chart-bearing table: (patient_id, <date_field>). Enables a "show me marker X over Y window" query without table scans.
  • Unique constraints on natural keys: (patient_id, day, source) for vitals_daily, (patient_id, ts) for glucose_points.
  • CHECK constraints for ranged numeric fields (mood_entries.valence -5..5, panic_events.severity 1..10).
  • Full-text search on writings.extracted_text via a Postgres tsvector + GIN index (declared in migration SQL, not in Drizzle).
  • Enums for closed sets (9 of them — see below); free text for sets with a long tail.

Structured vs. unstructured — where each kind of data ends up

Structured (most rows)

  • What: Plottable numerics, dates, foreign keys, free-text fields that are short and stable.
  • Where: Postgres columns with explicit types.
  • Examples: vitals_daily.hrv_ms, lab_results.value, medications.dose, glucose_points.mg_dl.
  • Query shape: SQL aggregates; never opens R2.

Semi-structured (JSONB)

  • What: Data with a stable envelope and a flexible payload — instrument-specific results, source-specific wearable extras, structured event metadata.
  • Where: Typed columns + a single jsonb column.
  • Examples: risk_assessments.payload (per-instrument scoring), wheel_of_life_assessments.scores (16 axes), vitals_daily.extras (per-source fields), audit_log.metadata (per-action context).
  • Query shape: SQL on the envelope; JSON path operators (->, ->>) on the payload when needed.

Unstructured (R2 blobs)

  • What: PDFs, DICOM bundles, audio, raw uploaded zips, ECG CSV exports, long-form patient writings.
  • Where: Cloudflare R2 object storage. Postgres holds the blob_key (single file) or blob_prefix (multi-file bundle).
  • Examples: documents.blob_key, imaging_studies.blob_prefix, writings.blob_key, ecg_events.blob_key.
  • Query shape: never queried by content from SQL. Worker reads bytes via env.R2_BUCKET.get(key).

Cloudflare R2 — the unstructured half

R2 layout · binding R2_BUCKET

  • Key shape: patients/<clerk_id>/<import_id>/<filename>
  • Per-study prefixes: imaging_studies.blob_prefix lets one row reference an entire DICOM bundle (many files).
  • Downloads: Worker streams bytes via signed responses; no R2 URL is ever exposed to the browser.
  • Cap on batch deletes: 1,000 objects per call — the GDPR-erasure routine pages through prefixes until empty.

How the bytes connect back

  • Single-file rows: blob_key column → one R2 object.
  • Multi-file rows: blob_prefix column → R2 list-objects under that prefix.
  • Provenance pointers: 14 clinical tables carry source_blob_key as an audit trail — the row was parsed out of this PDF.
  • JPEG previews: DICOM bundles get pre-rendered to PNG under jpeg_preview_prefix so the viewer doesn't need a DICOM decoder in the browser.
Schema source of truth. The full schema lives in db/schema.ts (Drizzle, 591 LOC). Migrations are version-controlled under db/migrations/. Local commands: npm run db:generate to diff the schema against migrations, db:migrate to apply, db:push for a fast schema sync against the dev database, and db:studio for a web UI inspector.
07 · AI layer

Three Claude models, three jobs

Different tasks have different latency and accuracy budgets. The platform uses three tiers of Claude — Opus for the user-facing chat, Sonnet for ingestion and dashboard composition, Haiku for high-volume classification. All three share the same Anthropic API endpoint and SDK; only the model identifier changes.

Model Identifier Use Called from Why this tier
Claude Opus 4.7 claude-opus-4-7 Patient-facing chat — questions about the record handleChat in web/_worker.js User-visible reasoning quality matters most. Strict system prompt: only answer from the bundled patient-record.txt; never invent facts.
Claude Sonnet 4.6 claude-sonnet-4-6 Lab PDF transcription, document ingest, dashboard composition lib/ingest.js, lib/dashboard.js Strong vision + structured-output reliability. Used for the heavy lifting that lands in Postgres.
Claude Haiku 4.5 claude-haiku-4-5-20251001 Document classification, reclassification, doctor-name backfill lib/ingest.js (taxonomy steps) High-volume, low-stakes calls. Latency and cost matter; accuracy is a bonus.
One key, three models. The Cloudflare Worker reads env.ANTHROPIC_API_KEY at request time. Each call site pins its own model identifier; there is no auto-routing or fallback cascade. If a tier is the wrong choice for a job, that is a code change, not a config change.
08 · Auth & access

Identity, sessions, role-based access

Authentication is handled by a custom cookie-session layer (lib/auth.js) modeled on Clerk's API surface. Three roles are defined in the user_role enum: admin, doctor, patient. Every API handler in the Worker calls authenticate(request, env) before touching any data; doctor-↔-patient access is gated by rows in patient_access.

Admin

  • Full read / write across every patient
  • Creates other users (doctors + patients)
  • Deletes patients (cascades through Postgres + R2)
  • Access: /api/admin/*

Doctor

  • Read access to patients they are linked to via patient_access
  • Can write dashboard sections + clinical notes
  • No create-user / delete-user rights
  • Access: /api/patient-* for linked patients only

Patient

  • Read access to their own record only
  • Can upload documents via the ingest path
  • Can wipe their own data (right to erasure)
  • Access: chat + their own static pages
09 · Deployment

Source vs. deploy — two separate operations

The GitHub repository is source of truth for code. It does not deploy anything. The live URL only updates when an engineer runs the Wrangler CLI against the local web/ folder. This separation is intentional — every deploy is an explicit, reviewed step.

10 · Security & compliance posture

Where we are, where we still need to get to

The architecture is shaped for eventual HIPAA / LGPD / GDPR compliance, but no formal certification is in place yet. Neon's EU region was chosen to keep data inside the most permissive jurisdiction; Cloudflare's Pages + R2 + Workers can all be configured for data-residency once the patient base grows.

In place today

  • EU-only Postgres region (Frankfurt)
  • Row-level patient ownership with ON DELETE CASCADE
  • R2 binding scoped to single bucket; per-patient key prefixes
  • Audit log of every mutation
  • Role-based API gating on every Worker handler
  • HTTPS-only · Cloudflare-managed TLS
  • Right-to-erasure: /api/patient-wipe-data cascades through Postgres + R2

Deferred until real patients

  • HIPAA Business Associate Agreement (BAA) with Cloudflare
  • HIPAA BAA with Anthropic (Enterprise plan)
  • Neon Enterprise plan (encryption-at-rest options)
  • SOC 2 / ISO 27001 attestations
  • Formal DPIA + LGPD Article 50 documentation
  • Penetration test on the Worker surface

Compensating controls

  • Patient writings are local-only until explicitly ingested
  • The 866 MB Apple Health XML is gitignored — never reaches the public repo or the deploy bundle
  • Wearable exports are kept off git for any non-Patient-Zero patient
  • Anthropic system prompt explicitly forbids inventing facts outside the record
Current posture in one sentence. Single-engineer, single-patient (plus three bespoke demo patients), pre-revenue, pre-certification — the stack is sized for <10 patients on Cloudflare's free / Pro plans plus a Neon Free or Launch plan. Upgrades to Enterprise tiers happen once real patients sign on and the compliance paperwork is signed.
