## 2. What's in an Oura export and where each part goes

First **detect the format** of the folder and log what you found:
- **Oura app export** — CSV "trends" (one daily row, many columns) and/or a JSON bundle.
- **Oura API v2 dumps** — JSON per endpoint: `daily_sleep`, `daily_readiness`, `daily_activity`,
  `sleep` (detailed sessions: hypnogram + `hr_5min` / `hrv_5min`), `heartrate` (continuous),
  `daily_spo2`, `daily_stress`, `daily_resilience`, `workout`, `sessions`, `tags`, `vO2max`.

Then route the data the platform's usual way:

| Oura data | Destination |
|---|---|
| **Raw export files** (the source CSV/JSON themselves) | **R2** under `patients/{patient_id}/oura/…`, with the usual pointer row(s) in `imports` / `import_files` |
| **Daily structured metrics** (sleep, readiness, activity, HR, SpO₂, temp, respiratory rate) | **Neon** — map onto existing `vitals_daily` columns |
| **Detailed sessions / continuous HR & HRV** | source material for the variability charts (§5) — only if the schema/scripts already consume it; otherwise keep the raw file in R2 and skip |

Concrete daily-metric mapping (map to whatever the columns are actually named in `schema.ts` —
add a migration only if a metric has **no** home, per §4):

- **Sleep:** total sleep time, sleep efficiency, deep / REM / light minutes, awake time,
  sleep latency, bedtime start/end, restfulness, sleep score. *(Sleep efficiency surfaces in
  the "At a glance" summary, so make sure it lands.)*
- **Readiness:** readiness score, **resting heart rate**, **HRV (rMSSD)**, body-temperature
  deviation, recovery index. *(Resting HR also surfaces in the summary.)*
- **Activity:** steps, active calories, total calories, MET minutes, activity score,
  sedentary time.
- **Other:** average SpO₂ (during sleep), respiratory rate, and stress/resilience if present.

## 3. FULL REPLACEMENT — the key requirement

Oura data **accumulates over time**: each export is a *superset* of every prior export. So the
correct, dedup-free strategy is **wipe the patient's existing Oura data, then load the new folder
fresh** — never merge or append.

Do this carefully and **scoped to (this patient) AND (Oura provenance) only**:

1. **Identify Oura provenance.** Determine exactly how an Oura-sourced row/blob is distinguished
   from Apple Health / Withings / lab data — a `source` column, the `import_source` enum value,
   or linkage through `imports` / `import_files`. **Use that as your delete filter.** If you
   cannot prove a row is Oura-sourced, do NOT delete it.
2. **Delete the old Oura data:**
   - The patient's Oura-derived rows in `vitals_daily` (and any sleep/HR detail tables fed from
     Oura).
   - The patient's Oura `import_files` + `imports` rows.
   - The R2 blobs under `patients/{patient_id}/oura/`.
3. **Guardrail — shared daily rows.** If `vitals_daily` stores **one row per day merged across
   devices** (rather than one row per source), do NOT nuke whole days. Instead null out only the
   Oura-owned columns for those days, then repopulate. Inspect the schema and pick the safe path;
   tell me which one you took and why.
4. **Never touch non-Oura sources.** Apple Health (incl. the ECG CSVs / export.xml), Withings,
   labs, imaging, writings, etc. must be left completely untouched.
5. **Then reingest** everything from the folder I confirmed in §0, fresh, for the patient I
   confirmed in §0.

Wrap the delete+reload in a transaction where feasible so a mid-run failure can't leave the
patient with half their Oura history.

## Provenance classification — device self-captured

This is a **device self-captured** source, so the clinician provenance facts do
**NOT** apply. Only the **measurement date** applies — it is the per-row date/timestamp
already written (`vitals_daily.day`, `ecg_events.recorded_at`) and the device is recorded

the existing `created_at`. Requesting/performing doctor and lab name/city/country are
`n/a` here — do **not** force the clinical fields onto wearable rows. (Contrast with the
clinician-ordered Blood/Imagery/ECG prompts, where all five facts apply.)
